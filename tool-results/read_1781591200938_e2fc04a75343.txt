     3→1.1 Project Summary
     4→Indicore Originals is a modern, mobile-first e-commerce platform selling curated products via Instagram-driven traffic. The MVP supports:
     5→ Cash on Delivery (COD) as the sole payment method.
     6→ Single-step checkout with integrated lead collection.
     7→ Product variants, image management, moderated reviews (registered customers only).
     8→ Guest order tracking (Order ID + Email).
     9→ Manual COD verification.
    10→ Homepage content management, FAQs, and analytics.
    11→Key Differentiators:
    12→ Single login page with role-based redirect (admin/customer).
    13→ Guest checkout with automatic order linking upon registration.
    14→ Purchase-validated reviews (only delivered orders can review).
    15→ Manual COD verification workflow (no OTP/SMS/email dependency).
    16→ Fixed 7-section homepage structure (content-managed, not structural).
    17→ All customer communication handled manually by admin via phone or WhatsApp.

   275→Rate Limiting Logic (Login - Database-based, no Redis):
   276→typescript
   277→// Database-based login rate limiting - no external dependencies
   278→async function checkLoginRateLimit(email: string): Promise<{ allowed: boolean; remainingMinutes: number }> {
   279→const customer = await db.customers.findUnique({ where: { email } });
   280→if (customer?.locked_until && customer.locked_until > new Date()) {
   281→const remainingMinutes = Math.ceil((customer.locked_until.getTime() - Date.now()) / 60000);
   282→return { allowed: false, remainingMinutes };
   283→}
   284→if (customer && customer.login_attempts >= 5) {
   285→await db.customers.update({
   286→where: { email },
   287→data: { locked_until: new Date(Date.now() + 15 * 60 * 1000) }
   288→});
   289→return { allowed: false, remainingMinutes: 15 };
   290→}
   291→return { allowed: true, remainingMinutes: 0 };
   292→}

  1375→Authentication Security
  1376→Requirement Implementation
  1377→Passwords hashed Supabase Auth (bcrypt, 10 rounds in auth.users)
  1378→No password_hash in customers table Supabase Auth is sole owner of passwords
  1379→JWT storage HTTP-only cookie, Secure, SameSite=Lax
  1380→Customer session 7 days (Supabase JWT expiry)
  1381→Admin session Manual logout only — no idle timeout
  1382→Password policy Admin: 16+ chars, unique; enforced in seed script
  1383→Initial admin Seed script only; no public admin registration route
  1384→Rate Limiting (Database-based - No Redis, No Upstash)
  1385→Endpoint Limit Window Storage Notes
  1386→Login 5 attempts 15 min customers.login_attempts / locked_until Per email
  1387→Order tracking 10 attempts 15 min order_track_attempts Per IP; rolling window
  1388→Review submission 5 submissions 1 hour Application logic Per customer
  1389→There is no Redis, Upstash, or any external caching service in V1. All rate limiting uses Supabase PostgreSQL.
  1390→

  1392→Row Level Security (RLS)
  1393→ Principle: Least privilege - every table has RLS enabled.
  1394→ Service role bypasses RLS for admin server-side operations only.
  1395→Critical: Guest INSERT policies - The following tables must have INSERT policies that allow unauthenticated users for guest checkout to
  1396→work:
  1397→ customers - allow INSERT for guest creation
  1398→ orders - allow INSERT for guest order creation
  1399→ order_items - allow INSERT for order item creation
  1400→ order_track_attempts - allow INSERT for rate limiting
  1401→Without these INSERT policies, guest checkout will fail with RLS violations.
  1402→Admin Route Protection (Middleware)
  1403→ Checks role in customers table on every /admin/* request.
  1404→ Redirects to /login if not authenticated or not admin.
